# T-058: LLM-based complexity estimation at plan time

- **Status**: TODO
- **Milestone**: 8 — Robustness
- **Depends**: none
- **PRD Reference**: §3.4

## Description

The current regex/heuristic complexity estimator (`src/ralph/core/complexity.ts`) misclassifies tasks because it relies on keyword matching and counting produces/dependencies. A task like "Test infrastructure — Docker lifecycle and per-test database isolation" that touches integration tests across 6 packages scored `standard` (75 turns) instead of `heavy` (125 turns) and ran out of turns at 76.

Replace the heuristic with an LLM call during `ralph plan` that reads the full task markdown and classifies complexity as `light`, `standard`, or `heavy`. Persist the result as a `- **Complexity**: <tier>` metadata field in the task file so the loop orchestrator reads it directly with zero runtime overhead.

## Requirements

1. **Add a `Complexity` metadata field to the task parser** — `src/ralph/core/tasks.ts` should parse `- **Complexity**: light|standard|heavy` from task files. If present, `scaleForComplexity` uses it directly and skips the heuristic.

2. **LLM classification during planning** — when `ralph plan` generates or updates task files, make a single cheap API call (Haiku-class model) per task with the full task markdown. The prompt should ask for a tier and one-line rationale. Write the tier into the task file's metadata block.

3. **Heuristic as fallback** — if no `Complexity` field is present (e.g. hand-written tasks), fall back to the existing regex heuristic so nothing breaks.

4. **`ralph estimate` subcommand** — a standalone command that re-estimates complexity for all TODO tasks and updates their `Complexity` fields. Useful for re-evaluating after task edits.

5. **All existing tests must continue to pass.**

## Produces

- Updated task parser to read `Complexity` field
- Updated `scaleForComplexity` to prefer explicit tier over heuristic
- LLM classification call integrated into `ralph plan`
- `ralph estimate` subcommand
- Tests for the new classification flow and fallback behavior

## Hints

- Use Haiku for the classification call — it's fast and cheap, and tier classification is a simple task
- The prompt should include the full task markdown and the tier definitions (turn/timeout limits) so the model can reason about scope
- Keep `computeTaskComplexity` as-is for the fallback path — just add a check for the explicit field before calling it
