# T-052: Fix monitor last output disappearing during tool-heavy stretches

- **Status**: TODO
- **Milestone**: 11 — Post-Iteration & Housekeeping
- **Depends**: T-051
- **PRD Reference**: §3.3

## Description

The monitor's "Last output" line disappears for long stretches when the agent is executing tool calls (reads, edits, bash commands) because most assistant messages during these stretches contain only `tool_use` blocks with no text. The `readLogTail` reads only the last 8KB, and `parseLastLogLine` finds no text content in that window, so `lastLogLine` is `null` and the line is omitted entirely. This makes it look like the agent is stuck.

**Requirements:**

1. **Staleness indicator on last output** — Always show the last output line, even if it was emitted a while ago. Include a staleness timer showing how long ago the text was emitted (e.g., `Last output (2m 13s ago): Let me verify the tests fail.`). This requires storing the timestamp of the log entry that produced the text. The line must never disappear — if no newer text is found on refresh, keep showing the previous text with an updated staleness timer.

2. **Activity indicator for tool calls** — When the most recent log entries are tool calls, extract the tool name (and optionally a target like the file path for Read/Edit or the command summary for Bash) and display it (e.g., `Activity: Bash (14s ago)` or `Activity: Edit src/ralph/core/process.ts`). This gives the user visibility into what the agent is doing between text outputs.

3. **Increase tail window** — Increase `readLogTail` default from 8KB to 32KB for the last-output scan so that text entries aren't pushed out of the window during long tool-call sequences. Alternatively, implement a two-pass approach: one scan for the last text entry, one for the last tool activity.

- All existing tests must continue to pass
- Add tests for staleness formatting, activity extraction, and the increased tail behavior

## Hints

- `extractTextFromJsonLine` already parses text from JSONL — extend or add a parallel function to also extract tool_use names and their inputs (the `name` field and optionally `input.file_path` or `input.command` truncated)
- For staleness, the timestamp now exists on each JSONL line (from T-051). When finding the last text entry, also capture its `timestamp` field. Compute staleness as `Date.now() - textTimestamp`.
- For the activity line, scan the tail for the last `tool_use` block — extract `name` and a short description from `input`
- The `MonitorData` interface needs new fields: `lastOutputTimestamp: Date | null`, `lastActivity: { tool: string; detail: string | null; timestamp: Date | null } | null`
- `formatMonitorOutput` should always emit "Last output" if there's text (with staleness), and always emit "Activity" if there's a recent tool call
- For the tail window increase, just change the default parameter in `readLogTail` or pass a larger value from `collectMonitorData`
