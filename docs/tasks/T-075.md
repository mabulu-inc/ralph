## Description

PRD §9 defines nine specialized agent roles (Product Manager, System Architect, Security Engineer, UX/UI Designer, Frontend & Backend Engineers, DevOps/SRE, SDET, Technical Lead, DBA/Data Engineer) that must participate in each iteration. Currently the system prompt treats the agent as a single undifferentiated developer. This task integrates the role-based collaboration model into the prompt templates so that each iteration produces structured, role-attributed commentary.

The roles participate at two gate phases:

- **Boot phase** (pre-implementation gate): PM, Architect, Security, DBA (if applicable), UX (if applicable) must each produce `[ROLE: ...]` commentary approving or adjusting the approach before any code is written.
- **Verify phase** (pre-commit gate): SDET (TDD compliance audit), Tech Lead (code review), Security (vulnerability scan), DevOps/SRE (CI/CD impact), DBA (if applicable) must each produce `[ROLE: ...]` commentary approving or flagging issues.

Roles not applicable to a task must explicitly state they are skipping with a reason (e.g., `[ROLE: DBA / Data Engineer] Skipping — no data models or persistence changes.`).

The SDET role during Verify has a specific mandate per §9.4: verify that TDD actually drove the development. Evidence of test-first discipline is required — tests that appear to have been written after the implementation must be flagged.

## Changes

### system-prompt.ts

Add the role definitions and participation rules to the system prompt template. This content is stable across iterations and benefits from prompt caching. Include:

- The nine role definitions with their focus areas (from §9.1)
- The participation-by-phase table (from §9.2) explaining which roles are active at Boot and Verify gates
- The `[ROLE: ...]` commentary format requirement (from §9.3)
- The TDD compliance audit criteria for the SDET role (from §9.4)
- Instruction that non-applicable roles must explicitly skip with a reason
- Instruction that gate phases (Boot and Verify) require all applicable roles to produce commentary before proceeding

### boot-prompt.ts

Add a `TASK CONTEXT FOR ROLES` section that helps roles determine applicability:

- Whether the task involves data models/persistence (signals DBA participation)
- Whether the task has user-facing surface — CLI output, error messages, interactive prompts (signals UX participation)
- The PRD sections referenced (already available as `{{task.prdReference}}`)

This section should use information already available from the task file — no new template variables needed.

### monitor.ts

The monitor already shows the latest text output via "Last output." When role commentary is the latest output, it will naturally appear. No structural changes needed to monitor — the `[ROLE: ...]` markers flow through the existing JSONL capture and text extraction.

However, consider detecting the `[ROLE: ...]` pattern in the last output line to show it more prominently, e.g., `Role: Technical Lead — Code structure is clean, naming is consistent...` instead of the raw marker. This is a nice-to-have, not a requirement.
