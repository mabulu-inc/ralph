# T-066: Fail-fast on quality-check preflight by default

- **Status**: TODO
- **Milestone**: 13 — Loop Efficiency
- **Depends**: none
- **PRD Reference**: §3.2
- **Complexity**: simple

## Description

Currently when `ralph loop` runs its quality-check preflight and finds pre-existing failures, it logs "pre-existing failures detected" and continues anyway. The baseline gets injected into the agent prompt as advisory context, but the loop still runs — meaning tasks can be marked DONE while the codebase is in a broken state.

The default should be fail-fast: if the quality check doesn't pass clean, abort the loop with a clear error message showing the failures. Add an `--allow-dirty` flag to opt into the current lenient behavior when the user explicitly wants to proceed despite pre-existing failures.

## Requirements

1. **Default fail-fast** — when `runPreflightCheck` returns `passed: false` (and not timed out), the orchestrator must abort with a non-zero exit (throw or `process.exitCode = 1`) and print the failures to stderr.

2. **`--allow-dirty` flag** — add to `LoopOptions` and `parseLoopOptions`. When set, use the current behavior: log "pre-existing failures detected", capture the baseline, and continue.

3. **Timeout behavior unchanged** — if preflight times out, proceed as today (log warning, no baseline).

4. **Dry-run unaffected** — `--dry-run` should not run the quality-check preflight (it already doesn't, just ensure no regression).

5. **All existing tests must continue to pass.**

## Produces

- Updated `LoopOptions` interface with `allowDirty` field
- Updated `parseLoopOptions` to handle `--allow-dirty`
- Updated orchestrator `execute()` to abort on preflight failure by default
- Tests for fail-fast behavior and `--allow-dirty` override
