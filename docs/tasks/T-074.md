## Description

When `ralph loop` hits its iteration limit (default 10), it prints only `"Loop complete"` — the same message it would print if all tasks were finished. This makes it impossible to determine why the loop stopped without manually inspecting task files. This was discovered in production use where ralph stopped twice while the developer was AFK, and they couldn't tell why it stopped. The core value proposition of ralph is unattended execution, so silent ambiguous exits undermine trust.

This task addresses three related gaps:

1. **Structured loop-end logging** — Write a `loop-end.json` file to `.ralph-logs/` when the loop exits, recording the reason, iteration counts, task counts, and spend. Replace the ambiguous `"Loop complete"` console message with a reason-specific message that includes remaining task counts and iterations used.

2. **Iteration visibility in monitor** — Expose the current iteration number and iteration limit in `loop-start.json` (or a new `loop-state.json` written each iteration) so that `ralph monitor` can display `Iteration: 5/10` (or `5/∞` when unlimited). The monitor dashboard should show this alongside existing progress info so the developer can see at a glance how much runway the loop has left.

3. **Dynamic task detection** — When the loop rescans tasks at the top of each iteration, detect when new task files have been added to `docs/tasks/` since the loop started (compare current total against the `loop-start.json` snapshot). Log a notice like `"[Iteration N] detected 3 new tasks (total: 26)"` and update the snapshot total so monitor progress bars stay accurate. This must be resource-efficient — it is already scanning tasks each iteration, so the detection should piggyback on that existing scan rather than adding filesystem watches or polling.

## Changes

### orchestrator.ts

- Add an `ExitReason` type: `'all_done' | 'iteration_limit' | 'budget_exceeded' | 'no_eligible_tasks' | 'user_interrupt'`
- At each exit point in `executeLoop`, write `loop-end.json` to `.ralph-logs/` with: `{ reason, endedAt, iterationsUsed, iterationsLimit, totalSpend, tasksCompleted, tasksRemaining, lastTaskId }`
- Replace `console.log('Loop complete')` (line 333) with a reason-specific message, e.g. `"Loop complete — iteration limit (10) reached, 13 TODO tasks remaining"` or `"Loop complete — all tasks done"`
- Write a `loop-state.json` at the start of each iteration with `{ iteration, iterationsLimit, currentTaskId, startedAt }` so monitor can read it
- After scanning tasks at the top of each iteration, compare `tasks.length` (total including DONE) against the snapshot total. If new tasks detected, log a notice and update `loop-start.json` total field

### monitor.ts

- Read `loop-state.json` to get iteration number and limit
- Add an `Iteration: 5/10` (or `5/∞`) line to the dashboard output
- Read `loop-end.json` when status is STOPPED to show the exit reason, e.g. `Stopped: iteration limit reached (10/10), 13 tasks remaining`
- Add `iterationInfo` and `exitReason` fields to `MonitorData`

### Types and interfaces

- Add `LoopEndSnapshot` and `LoopStateSnapshot` interfaces (can live in monitor.ts or a shared types file, follow existing patterns)
